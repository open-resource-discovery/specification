The Problem

Without ORD

  • Metadata scattered across systems
  • Manual documentation maintenance
  • Point-to-point integrations
  • Metadata drift over time

Discovery Problem

  • Which events are published?
  • What data models exist?
  • No single source of truth
Documentation Problem
  • Outdated API catalogs
  • Manual sync processes
  • Inconsistent formats
  • Multiple tools per standard
Runtime Problem
  • Tenant-specific configs unknown
  • No dynamic discovery
  • Custom deployments invisible

Tooling Problem
  • Each tool needs custom adapter
  • AI tools can't auto-integrate
  • Analytics require manual setup
  • Low-code tools limited

Result: High integration costs, slow time-to-market, brittle connections

What is ORD?

A protocol that allows applications to self-describe their resources

  • One standard for APIs, Events, Data Products, and more
  • Machine-readable metadata in JSON format
  • Decentralized - applications are the source of truth

Data Model Deep Dive

â†“ ID System & Grouping

ORD ID System

Namespaces (Hierarchical) 

vendor:product:sub-context
Example: sap.s4:communicationScenario:
  • Vendor: Organization identifier (e.g., sap, microsoft)
  • Product: Product/system identifier (e.g., s4, btp)
  • Sub-context: Optional refinement (e.g., communicationScenario)
ORD ID (Globally Unique) 

namespace:type:localID
Example: sap.s4:apiResource:BusinessPartner:v1

Used for: Packages, APIs, Events, Entity Types, etc.

Other ID Types
  • Correlation ID: Reference resources across systems
  • Specification ID: Reference standards/strategies
  • Local ID: Unique within namespace

Grouping & Bundling Concepts

Packages (Mandatory)
  • Top-level organization unit
  • Every resource belongs to exactly one package
  • Can represent: products, modules, features
  • Metadata: title, description, vendor, version
  • Publishing unit for ORD aggregators
Consumption Bundles
  • Technical grouping by credentials
  • Resources sharing same authentication
  • Reduces integration complexity
  • One token = access to all in bundle
  • Links to access strategies
Groups (Custom Types)
  • Flexible custom groupings
  • Examples: business scenarios, capabilities
  • Extensible group type system
  • Multiple groups per resource
  • Semantic relationships
Tags & Labels
  • Free-form categorization
  • Industry tags (e.g., finance, healthcare)
  • Technical tags (e.g., beta, premium)
  • Custom key-value labels
  • Supports faceted search

Perspectives: Static vs Dynamic

Static Perspective (system-version)
  • Generic, design-time view
  • Product-level metadata
  • What could be available
  • Version-specific capabilities
  • Use case: API catalogs, documentation

Example: "S/4HANA Cloud v2408 supports 500 APIs"

Dynamic Perspective (system-instance)
  • Runtime, instance-specific view
  • Tenant-level configuration
  • What is actually available
  • Custom extensions included
  • Use case: System landscapes, integration

Example: "Customer X's S/4HANA has 327 active APIs + 15 custom"

â†“ Perspective Use Cases

When to Use Each Perspective

Use Case Perspective Why
API Catalog Static Show all possible APIs for a product version
System Landscape Dynamic Show actual deployed systems and configs
Integration Setup Dynamic Need tenant-specific endpoints and auth
Product Documentation Static Document standard product capabilities
Pre-sales Evaluation Static Understand out-of-box features
Custom Extension Discovery Dynamic Custom APIs only exist at runtime
AI Agent Auto-Integration Dynamic Need actual available APIs to call

Best practice: Maintain both perspectivesâ€”static for documentation, dynamic for runtime

â†“ Versioning & Extensibility

Version Management in Detail

Semantic Versioning 

MAJOR.MINOR.PATCH
  ^     ^      ^
  |     |      +-- Bug fixes, backwards compatible
  |     +--------- New features, backwards compatible
  +--------------- Breaking changes, not backwards compatible
Lifecycle States

Release Status:

  • active - Production ready
  • beta - Preview, may change
  • deprecated - Marked for removal

Deprecation Info:

  • Deprecation date
  • Removal/sunset date
  • Migration guidance
  • Successor reference
Version Evolution

Resources can link to predecessors/successors, enabling automated migration path discovery
